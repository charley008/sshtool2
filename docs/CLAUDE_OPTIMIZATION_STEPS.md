# xplot.sshtools-2.0.0 后续优化步骤

本文档用于指导 Claude Code App 对 `xplot.sshtools-2.0.0` 做分阶段优化。当前插件已经可以正常使用，所以后续目标不是推倒重写，而是保持原有功能可用的前提下，逐步补齐稳定性、安全性、界面一致性和发布质量。

## 总原则

每一轮只处理一个明确范围，不要一次性要求 Claude “优化整个插件”。

推荐每次都用下面这段作为任务前缀：

```text
这是 VS Code 插件 xplot.sshtools-2.0.0，是我根据 xplot.sshtools-1.0.81 逆向后重写的源码版。
要求：
1. 优先保持和 1.0.81 行为一致
2. 不要大重构
3. 每次只处理一个明确范围
4. 修改前先说明会改哪些文件
5. 不要删除现有功能
6. 改完必须运行 npm run build
7. 如果发现风险，按严重程度列出
8. 所有改动都要兼容已有用户配置
本轮任务是：……
```

## 阶段 1：建立差异基线

先让 Claude 只分析，不修改代码。

```text
你先不要改代码。请分析 xplot.sshtools-2.0.0 当前源码，和 xplot.sshtools-1.0.81 的 package.json / out 行为 / 资源结构做对比。
目标：
1. 找出 2.0.0 与 1.0.81 在命令、菜单、视图、webview 页面、资源路径上的差异。
2. 找出可能导致功能缺失或 UI 不一致的点。
3. 按严重程度排序输出，不要修改文件。
```

输出结果建议保存下来，作为后续每轮修改后的对照基线。

## 阶段 2：生成回归测试清单

插件缺少大量用户反馈时，先用人工测试清单模拟反馈。

```text
请基于当前插件功能，写一份详细的人工回归测试清单，不要改代码。
覆盖：
1. SSH 新增、编辑、删除、连接、断开、终端
2. FTP 新增、编辑、删除、上传、下载、打开文件、保存回传
3. 工作区添加、删除、重命名
4. 分组筛选、重命名、移入、移出
5. 端口转发、Socks5、远程桌面
6. 配置导入导出、剪贴板导入
7. Windows 路径、空格路径、中文路径、大文件、取消操作、网络超时
每个用例写：操作步骤、预期结果、可能风险。
```

后续每修完一批问题，至少跑一遍常用 SSH/FTP 流程。

## 阶段 3：先修高风险稳定性问题

第一轮实际改代码时，只修稳定性，不碰界面，不改功能表现。

```text
请只修复高风险稳定性问题，不要重构 UI，不要改变功能表现。
优先处理：
1. showInputBox 用户取消时 input 为 undefined 导致 trim 报错
2. setInterval 没有保存和 deactivate 清理
3. SSH/FTP 连接 end/error 回调访问已删除 activeConn 导致异常
4. 文件上传下载 Promise 过早 resolve，应该监听 finish/error
5. 已贡献但未注册的命令，例如 xplot.terminal.hear
要求：
- 每个改动都说明修改文件和原因
- 尽量保持原版行为
- 改完运行 npm run build
```

这一阶段完成后，重点测试：

- SSH 新增、测试连接、保存、打开终端
- FTP 新增、打开文件、保存回传
- 用户在所有输入框按 Esc 取消
- 下载和上传小文件、大文件

## 阶段 4：命令和生命周期清理

如果阶段 3 没有完全覆盖，可以单独补一轮。

```text
请检查 VS Code 命令注册和扩展生命周期。
目标：
1. package.json contributes.commands 中的命令都应该能注册或说明为什么不注册
2. deactivate 时清理定时器、连接、webview 相关资源
3. 不新增重复定时器
4. 不改变命令 ID
5. 改完运行 npm run build
```

## 阶段 5：文件传输可靠性

文件传输是 SSH/FTP 工具最容易产生真实用户反馈的地方，建议单独处理。

```text
请重点检查 SSH/FTP 文件上传、下载、打开远程文件、本地保存回传逻辑。
要求：
1. Promise 应该在真实写入完成后 resolve
2. 正确处理 stream error、finish、close、取消操作
3. 不要显示成功但文件实际不完整
4. 保持现有 UI 和提示文案基本不变
5. 改完给出手工验证步骤，并运行 npm run build
```

测试时建议覆盖：

- 小文本文件
- 10MB 以上文件
- 中文文件名
- 空格路径
- 上传中取消
- 下载中断网或断开 SSH/FTP

## 阶段 6：安全加固

安全相关可能影响功能，必须单独一轮做。

```text
请做安全加固，但保持现有用户数据兼容。
重点：
1. 检查 child_process exec/execSync 拼接命令的位置
2. 对 host、username、password、port、path 等参数做安全处理
3. 能用 spawn/execFile 参数数组的地方替换 exec 字符串拼接
4. 不要破坏 Windows 端口转发、RDP、VNC 的现有使用方式
5. 如果某些地方暂时不能安全替换，请标注 TODO 和风险
改完后给出手工验证步骤。
```

注意：凭据存储不要和命令注入一起大改。`SecretStorage` 迁移建议作为后续独立任务，因为它涉及老用户配置兼容。

## 阶段 7：凭据存储迁移

这一阶段可选，但如果准备长期维护或发布，建议做。

```text
请设计 SSH/FTP 密码、私钥、passphrase 从 globalState 迁移到 VS Code SecretStorage 的方案。
先不要改代码，先输出方案：
1. 当前凭据保存在哪里
2. 如何兼容已有用户配置
3. 首次读取旧配置时如何迁移
4. 导入导出时如何处理敏感字段
5. 失败回滚方案
6. 需要修改的文件列表
```

方案确认后再执行：

```text
请按上一步方案实现 SecretStorage 迁移。
要求：
1. 兼容已有 globalState 配置
2. 不丢失用户已有连接
3. 导入导出行为要明确说明是否包含密码/私钥
4. 改完运行 npm run build
5. 给出迁移测试步骤
```

## 阶段 8：对齐原版界面

界面不要直接说“优化一下”，要先让 Claude 对比原版。

```text
请对比 1.0.81 与 2.0.0 的 webview 和资源表现，目标是让 2.0.0 尽量接近原版界面。
先不要改代码，先输出：
1. 原版有哪些页面：SSH 编辑、FTP 编辑、配置管理、转发管理、远程桌面、终端
2. 2.0.0 当前页面与原版可能不同的地方
3. 哪些差异会影响用户使用，哪些只是视觉差异
4. 建议分几批改，每批改哪些文件
```

确认后按页面拆分，例如先改 SSH/FTP 表单：

```text
请只优化 SSH/FTP 新增编辑页面，使它更接近 1.0.81 原版体验。
要求：
1. 不改后端存储结构
2. 不改命令 ID
3. 保留现有保存/测试事件
4. 修复乱码文案和布局问题
5. 改完运行 npm run build
```

再分别处理：

- 配置管理页面
- 端口转发页面
- 远程桌面页面
- 工作区页面
- 终端页面

## 阶段 9：修复中文乱码和文案

文案建议单独修，避免和业务逻辑混在一起。

```text
请修复 package.nls.zh-cn.json 和 webview 中明显乱码的中文文案。
要求：
1. 保持 key 不变
2. package.nls.json 英文不动，除非明显拼写错误
3. webview 中硬编码中文改成清晰简体中文
4. 不改业务逻辑
5. 改完确认 package.nls.zh-cn.json 仍是合法 JSON
6. 改完运行 npm run build
```

修完后重点检查：

- 活动栏标题
- 右键菜单
- 输入框 placeholder
- 成功/失败通知
- webview 表单 label
- 终端设置面板

## 阶段 10：发布前检查

最后让 Claude 执行能自动执行的检查，并列出人工测试项。

```text
请为 xplot.sshtools-2.0.0 写发布前检查清单，并执行能自动执行的部分。
检查：
1. npm run build
2. package.json contributes 命令是否全部注册
3. resources 图片是否存在
4. out/webview html/js 是否生成
5. package.nls*.json 是否能 JSON.parse
6. 搜索明显 console.log 调试残留
7. 搜索 TODO/FIXME/throw new Error(e)
8. 列出还需要人工测试的项目
```

## 推荐执行顺序

1. 只分析差异，不改代码
2. 生成测试清单
3. 修稳定性 bug
4. 修命令和生命周期
5. 修文件传输可靠性
6. 修安全风险
7. 设计并迁移凭据存储
8. 对齐原版 UI
9. 修中文乱码
10. 发布前检查
11. 打包 `.vsix` 后本机安装测试

## 每轮修改后的固定检查

每轮改完后至少执行：

```powershell
npm run build
```

并人工确认：

- 插件能激活
- 活动栏两个入口存在
- SSH/FTP 列表能刷新
- 新增连接页面能打开
- 保存连接后数据不丢
- 右键菜单没有明显失效项
- 输出面板没有新的异常日志

## 注意事项

- 不要让 Claude 一次性大规模重构。
- 不要先做界面，再做稳定性。
- 不要把安全加固和 UI 对齐混在同一轮。
- 不要轻易改变配置结构，除非有迁移方案。
- 不要删除旧功能，即使当前看起来暂时没用。
- 每轮都保留构建输出和修改说明，方便回滚。
